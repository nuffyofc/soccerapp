### Key decisions made
- **Positioning**: soccer/World Cup-first, zero-stats, pure-preference swipe game — not a stats app, not a trivia app.
- **Multi-category architecture**: one app, not separate apps per sport/category. Confirmed via competitive-analysis reasoning — shared voting engine, share-card generator, and Hot Debates logic mean forking into separate apps would just multiply acquisition cost with no product benefit.
- **MVP feature set locked**: swipe/tap vote → animated % reveal → auto-advance → 10-round session → shareable result card, plus the Hot Debates leaderboard (reuses existing vote data, no new backend needed).
- **Explicitly deferred to Phase 2**: trivia mode (correct-answer quiz), daily crossword (Wordle-style daily ritual game), personal streak tracking. All three are separate game mechanics from the core swipe loop and would roughly double MVP build scope — flagged as a standing content-ops commitment for the crossword specifically (needs a permanent daily-puzzle pipeline, not a one-time build).
- **Paywall**: none at MVP launch. Core loop (swipe, vote, % reveal, basic share card) is permanently free — it's the growth engine. First monetization touchpoint only after real usage data exists.

### Working demos built (HTML/CSS/JS, self-contained, no backend)
Three files exist in this folder, functionally tested in-browser (swipe/tap vote, animated % reveal, auto-advance, 10-round session, share screen, Hot Debates leaderboard all confirmed working):

- **[demo-soccer.html](demo-soccer.html)** — soccer/World Cup only, no category switcher. This is the "clean MVP" reference build: 20 players across Argentina/France/England/Brazil/Portugal, rivalry-weighted pairing (Messi/Ronaldo etc.), votes persisted to `localStorage` to simulate live crowd data.
- **[demo-allsports.html](demo-allsports.html)** — same engine plus a burger-menu category switcher (Soccer ⚽ / Basketball 🏀 NBA / Esports 🎮 CS2 / Movies 🎬 Actors). Each category has its own curated player pool and rivalry pairs; votes are namespaced per category in `localStorage` so stats don't mix across categories.
- **demo.html** — identical duplicate of `demo-allsports.html` (predates it; kept only for backward compatibility since it was shared earlier in the session — candidate for deletion to avoid confusion, not yet removed).

All three are self-contained single-file HTML (no build step, no server dependency) — open directly in any browser. Voting is simulated via `localStorage` (no real backend yet — see Phase 1 build plan below).

### Infra check
- Confirmed the app can be served over LAN (tested via `python -m http.server` bound to `0.0.0.0`, accessed from another device on the same Wi-Fi at `http://192.168.1.12:8000/...`). Server was stopped after testing — not currently running.

## 2. Full Plan Going Forward

### Phase 0 — MVP validation (current phase)
**Goal**: prove the swipe → % reveal → share loop actually gets shared and drives return visits, before investing in more content or infrastructure.
- [ ] Decide: native app (React Native/Expo) or mobile web first — mobile web is faster to ship for validation; native helps app-store discovery long-term (see [PLAN.md](PLAN.md) open questions)
- [ ] Stand up real backend (Supabase: `votes(pair_key, winner_id, count)` table, normalized pair ordering) to replace the `localStorage` simulation — this is the main gap between the current demo and a real MVP
- [ ] Build real image-export for the share card (client-side canvas/html-to-image) — currently a placeholder `alert()` in the demo
- [ ] Finalize the soccer-only player roster and rivalry-pair seed list for actual launch (demo data is a placeholder set)
- [ ] Ship soccer-only (per [PLAN.md](PLAN.md) "Out of Scope for MVP" — multi-category is Phase 2 even though the demo already has 4 categories built for internal comparison/pitching purposes)
- [ ] No paywall at launch (per [STRATEGY.md](STRATEGY.md) §4)

### Phase 1 — Real backend + polish
- Replace `localStorage` vote simulation with live Supabase-backed voting
- Implement close-split (40-60%) pairing weighting once real vote volume exists (currently only rivalry-tagged weighting is implemented, per [STRATEGY.md](STRATEGY.md) §3)
- Apply full [DESIGN.md](DESIGN.md) visual polish pass: typography (condensed display font), sound design, refined motion/haptics
- Real share-card image export tuned for IG Stories/TikTok/YT Shorts (9:16 native)

### Phase 2 — Expansion (post-validation only)
- Multi-category rollout (Basketball/NBA, Esports/CS2, Movies/Actors) — architecture already proven in `demo-allsports.html`
- Trivia mode (separate mechanic, own dataset with correct answers + scoring)
- Daily crossword (separate mechanic, needs a standing daily-content pipeline — decide content source before committing)
- Cross-category discovery, community-submitted matchups, site-wide leaderboard expansion
- First paywall touchpoint: soft share-card style upsell, then subscription for extra categories/stats after 3-5 sessions of demonstrated engagement (see [STRATEGY.md](STRATEGY.md) §4)
